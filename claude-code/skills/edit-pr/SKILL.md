# Edit PR

PR의 타이틀과 바디를 작성한다.

## Trigger

`/edit-pr` 또는 PR 타이틀/바디 작성 요청 시.

## Instructions

1. `gh pr view <PR번호> --json title,body,headRefName,baseRefName` 으로 현재 PR 정보를 가져온다.
2. `gh pr diff <PR번호>` 으로 변경사항을 확인한다.
3. 기존 바디의 **템플릿 원본을 절대 수정·삭제하지 않는다**. 체크박스, 섹션 헤더, 구분선, 안내 문구 등 템플릿 구조를 그대로 유지하고, 해당하는 체크박스만 `[x]`로 체크한다. 단, "PR 제목 예시" 섹션은 제거한다.
4. 각 섹션을 채운다:
   - **PR 유형**: 해당하는 항목 체크
   - **Notion Issue Card Link**: 유저가 제공하지 않으면 비워둔다
   - **작업 범위**: 해당하는 항목 체크. "기타"인 경우 괄호 안에 직접 입력
   - **기능 요약**: 변경사항을 한 문장으로 요약
   - **주요 변경사항**: 단순 "무엇을 바꿨는지" 나열이 아니라, "왜 그렇게 했는지" 맥락을 포함한다. 리뷰어가 diff를 안 봐도 설계 의도를 파악할 수 있어야 한다. 변경을 테마별로 묶고 각 테마를 `#### 제목4`로 쓴 뒤 그 아래에 bullet을 나열한다. sub-bullet을 중첩하지 말고 flat bullet으로 유지한다.
   - **테스트 방법**: 변경사항 검증 방법
   - **기타 참고 사항**: 관련 PR, 후속 작업, 주의사항 등. 선행/후속 PR이 있으면 번호와 관계를 명시한다. PR 링크는 `[#179](https://github.com/<owner>/<repo>/pull/179)` 형식으로 축약한다.
6. 기술적 상세 정보(검증 결과, 파일 목록, 비교 데이터 등)는 `<details><summary>...</summary>` 접기 블록으로 감싼다. 본문은 깔끔하게 유지하되 필요한 정보는 접기 안에 넣는다.
7. `gh pr edit <PR번호> --title "..." --body "$(cat <<'EOF' ... EOF)"` 로 업데이트한다.

## Rules

- 템플릿 원본의 어떤 요소도 삭제·수정·재구성하지 않는다. 내용을 채워넣기만 한다.
- 유저가 명시한 워크스코프, 키워드를 그대로 반영한다.
- 타이틀 형식: `feat:`, `fix:` 등 conventional commit prefix + `(프로젝트명)` + 요약.
- 내용은 간결하게 핵심만 쓴다. 장황한 설명, 불필요한 수식어, 반복 서술 금지.
- 마크다운 서식을 과도하게 사용하지 않는다 (볼드, 코드블록 등 최소화). 단, 유저가 테이블 등 특정 서식을 요청한 경우는 예외.
- 변경사항은 "what + why" 구조로 쓴다. 예: "v5 응답을 기존 v4 구조로 변환하여 기존 코드에 영향 없이 전환" (O) vs "변환 함수 추가" (X)
- **한 bullet = 한 문장 = 한 줄**. 쉼표·점으로 두세 문장을 이어붙이지 않는다. 한 bullet이 대략 80자를 넘거나 쉼표로 이어지는 절이 2개를 넘으면 별도 bullet으로 분해한다.
- **주요 변경사항**은 `#### 테마 제목` + 그 아래 flat bullet 구조로 쓴다. bullet 중첩(sub-bullet) 금지.
- `####` 섹션 사이, 그리고 `####` 제목과 첫 bullet 사이에 **빈 줄 하나**를 넣어 시각적 그룹을 분리한다. 빈 줄 없이 붙이면 벽처럼 보여 가독성이 죽는다.

예시:

나쁜 예 (한 줄에 절 3개 + 중첩):

```
- 신설: userId 해시 기반 팔레트 매핑을 통합, indicator/row-viewer에 중복 구현돼 있던 HSL 해시가 다른 색 공간을 쓸 위험이 있어, 팔레트 10종으로 축소
```

좋은 예 (테마별 ####  + flat bullet):

```
#### 색상 모듈 단일화

- `presence-color.ts` 신설로 userId 해시 → 팔레트 매핑 통합
- 기존엔 indicator/row-viewer 두 곳에 HSL 해시가 중복 구현돼 색 공간이 어긋날 위험
- 고정 팔레트 10종으로 축소해 동일 user는 어디서든 같은 색쌍 보장

#### CSS 변수 기반 반환

- 반환을 `{ "--border", "--dot" }` 객체로 설계
- SCSS에서 `var(--border)` / `var(--dot)`로 소비
```
